    # GRASP

    • Information expert
      • 信息不单指数据
      • 谁具有实现职责必要的信息谁就是专家，并把职责赋给它
      • 使用自身信息完成职责使得维持封装性、不增加耦合度，行为分布在所需信息的类之间使得功能集中、提高内聚性
    • Creator
      • 首选，b聚合/组合a
      • b记录a
      • b频繁使用a
      • b具有a的初始化数据且在创建a时传递给a
    • Controller
      • 外观控制器的所有变体：代表整个系统、跟对象、运行软件的设备或主要子系统
      • 代表用例场景且发生系统事件
    • Indirection
      • 为避免直接耦合，将职责分配给中介，支持防止变异
    • Low coupling
      • 对元素职责集中度的评估；内聚过高，不相关功能在一个模块，耦合度高，发生变更影响不可控
      • 去除的是不良耦合的负面情况：难以理解、变化会影响依赖不稳定元素的元素
        • 分配职责，评估可选方案
        • 耦合不能独立考虑
        • 修改模块后的涟漪效应，通常是低内聚的原因
        • 模块间的依赖使组合困难，可复用性低，通常是耦合高的原因
      • 方向
        • 循环依赖（更差）：链路更长，范围更大
        • 双向依赖（差）：相互依赖的元素不能独立行动
        • 单向依赖（好）
      • 链路
        • 深度
        • 广度
        • 不加约束的链路变宽会产生上帝元素，容易产生双向循环
      • 方式：
        • 内容耦合 content（高）：模块A直接使用模块B的内部数据或通过非正常入口进入模块内部
        • 共享耦合/公共耦合 common（高）：通过公共数据环境相互作用，模块越多越复杂
        • 控制耦合 control（中）：模块A调用模块B时传递控制变量（开关、标志等），模块B通过控制变量有选择执行功能
        • 特征耦合/标记耦合 stamp（中）：多个模块共享复杂数据结构（数组、记录、文件），传递的是数据结构的地址
        • 数据耦合 data（低）：模块借传入值共享数据，每个数据是基本数据且只分享这些数据（传递整数到数学函数）
        • 非直接耦合 message（低）：模块间无直接关系，通过主模块控制调用，独立性强
        • 无耦合（无）：模块完全不合其他模块交换信息
      • 降低耦合方法：
        • 依赖方向：使用单向依赖，去除/弱化双向依赖，不使用循环依赖
        • 依赖链路：遵守最少认知原则
        • 依赖方式：尽量数据耦合，少用控制和特征耦合，控制公共耦合范围，不使用内容耦合，如果依赖对象不稳定则使用非直接耦合弱化耦合紧密度
        • 分配正确职责：信息专家、创建者
        • 通过其他原则减少不稳定元素的影响
    • High cohesion
      • 对元素职责集中度的评估；内聚过高，不相关功能在一个模块，耦合度高，发生变更影响不可控
      • 结构化 -> 增加抽象层侧 -> 保持高内聚、低耦合
      • 怎样保持对象是有重点的、可理解的、可维护的、支持低耦合的
        • 按照相关性分配职责
        • 分解后的元素更加简单易于理解维护
        • 按照相关性拆分提高重用性
      • 低内聚的类需要做不相关的工作或大量工作
        • 难以理解
        • 难以复用
        • 难以维护
        • 常受变化影响
      • 内聚程度由低到高
        • 偶然 coincidental：模块功能恰好在一起，各个功能只是同在一个模块
        • 逻辑 logical：不管本质，只要功能在逻辑上是同类就在一起
        • 时间 temporal：时间点相近的程序在一个模块（异常 -> 回调 -> log -> 通知）
        • 程序内 procedural：按照固定顺序执行的功能在一个模块（权限 -> openFile）
        • 通信 communicational：处理相同数据功能在一个模块
        • 顺序 sequential：模块各功能彼此I/O相关，有响应地先后顺序
        • 功能 functional： 模块中各功能对模块单一明确定义的任务有贡献（xml词法分析）
    • Polymorphism
      • 当相关选择、行为根据类型变化，则用多态为变化的行为类型分配职责
      • 强扩展
      • 客户端无感
    • Protected variations
      • 识别预计的变化，围绕变化分别职责创建稳定接口
    • Pure fabrication
      • 对不代表问题域的纯虚构的类分配高内聚职责，用以支持高内聚/低耦合/复用
      • 所有设计模式本质是纯虚构
      • 接纳问题专家所分配给领域类的职责
      • 创建与业务更相近的纯虚构才能更容易理解和管理耦合关系
